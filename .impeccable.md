## Design Context

### Users

Two audiences, weighted equally:

1. **Developers** — built the Next.js site, drop in the CMS package, occasionally edit content themselves. Technical, comfortable with dark UIs and dense information. They want correctness and zero surprise.
2. **Content editors** — a client or teammate who didn't write the code. Uses the admin panel regularly to update pages. Needs clear affordances, low friction, and confidence that their actions are reversible.

The UI must be legible to a non-technical editor on first contact while remaining efficient for a developer in flow. Avoid jargon in labels and copy. Every destructive action needs clear confirmation.

### Brand Personality

The CMS has no brand of its own — it lives inside someone else's product. Its visual defaults should recede, not compete. The personality is:

- Quiet confidence — polished enough to feel trustworthy, neutral enough not to impose
- Focused — nothing decorative that isn't functional
- Reliable — feedback is immediate, states are clear, errors are human-readable

Three-word personality: **quiet, polished, trustworthy**

Emotional goal: editors feel in control. Developers feel nothing is fighting them.

### Aesthetic Direction

- Dark mode only for now (light mode planned, stub already in CSS)
- Surface: near-black (`#191919`), card: `#252521` — warm dark, not cold gray
- Accent: keep neutral defaults so consumer theming is the star. The current blue (`#4a90d9`) should function as a safe fallback, not a brand color. Prefer lower-saturation or near-neutral accents when adding new tokens.
- Typography: system font stack at 0.875rem — compact, readable, no typographic personality
- Animation: subtle, purposeful. Framer Motion is already a peer dep — use it for drag reordering and toasts, but not for decoration
- Reference feel: Ghost CMS, Linear, Vercel dashboard — product-grade without being showy
- Anti-reference: WordPress admin, builder tools with lots of chrome, anything that feels like "a CMS"

### Design Principles

1. **Recede from the consumer's brand** — defaults are neutral; every color token is overridable. The admin UI should feel like it belongs in any Next.js project.
2. **Confidence through clarity** — status is always visible (draft vs published, auto-save indicator, loading states). Editors never wonder if their action worked.
3. **Polished, not decorative** — transitions and motion exist to orient the user (drag, toast, modal), not to impress. Every animation earns its place by reducing cognitive load.
4. **Zero friction for both audiences** — non-technical editors get plain-language labels and forgiving flows; developers get keyboard-operable controls and predictable behavior.
5. **Accessibility is structural, not cosmetic** — WCAG AA is already enforced via hooks and CI. Design decisions must not trade accessibility for aesthetics. Focus rings, touch targets, and live regions are non-negotiable.
