<objective>
Wire locale awareness into the content read path and the front-end block renderer. All six primary block components (Title, Paragraph, RichText, Image, Video, Gallery) accept a `locale` prop and resolve locale-object fields through the `resolveLocale` utility created in Plan 02.

Purpose: This is the consumer-facing runtime layer for bilingual content. D-06 demands that existing callers (no options arg, plain string fields) continue to work identically — the locale parameter is purely additive, and `resolveLocale` passes plain strings through unchanged.
Output: 2 entry files updated, 1 block assembly updated, 6 block components updated.

```js
// src/utils/resolveLocale.js
export function resolveLocale(value, locale = 'en') {
  if (value === null || value === undefined) return ''
  if (typeof value !== 'object') return value
  return value[locale] ?? value['en'] ?? ''
}
```

Localizable field map (confirmed by code inspection):

| Block     | Localizable fields               | Non-localizable fields         |
|-----------|----------------------------------|--------------------------------|
| Title     | data.text                        | data.level                     |
| Paragraph | data.text                        | —                              |
| RichText  | data.html                        | —                              |
| Image     | data.alt, data.caption           | data.src, data.width, data.height |
| Video     | data.title (caption text)        | data.url, data.src             |
| Gallery   | items[].alt, items[].caption     | items[].src, items[].id        |

The Blocks renderer forwards `locale` to EVERY block component (including List and PullQuote, even though Plan 11's scope covers the six primary block types). List and PullQuote are NOT modified in this plan — they continue to ignore the `locale` prop they receive, which is safe because unused props are discarded in React. Adding resolveLocale to List/PullQuote is a post-v1 concern.

WARNING: The `data.html` field in RichText is passed through `dangerouslySetInnerHTML` after sanitization. When localized, `resolveLocale(data.html, locale)` returns a plain string that flows through the same sanitizer — no change to the sanitization path.
