Add __session cookie lifecycle to useAuth hook, enabling withCMSAuth middleware to read auth state from cookies on server-side requests.

This plan bridges the gap by writing and clearing the cookie inside the onAuthStateChanged callback.

  <action>
Modify the `useAuth` function in `src/index.js`. The only change is to the `subscribeToAuthState` callback (lines 35-38). Replace the synchronous callback with an async one that manages the `__session` cookie:

```javascript
const unsubscribe = subscribeToAuthState(auth, async (u) => {
  if (u) {
    const token = await u.getIdToken()
    const secure = typeof document !== 'undefined' && document.location.protocol === 'https:' ? '; Secure' : ''
    document.cookie = `__session=${token}; path=/; SameSite=Strict${secure}`
  } else {
    document.cookie = '__session=; path=/; SameSite=Strict; max-age=0'
  }
  setUser(u)
  setLoading(false)
})
```

CRITICAL ordering: `await u.getIdToken()` MUST complete before `setUser(u)` and `setLoading(false)`. This ensures the cookie is written before React re-renders and any downstream component reads the cookie.

The `typeof document !== 'undefined'` guard on the Secure flag prevents SSR crashes (though useAuth is client-only, defensive coding is cheap).

Do NOT:
- Set the cookie in LoginPage after signIn() — onAuthStateChanged fires automatically
- Import anything from 'firebase/auth' directly — use existing subscribeToAuthState
- Call getIdToken(true) (forced refresh) — normal getIdToken() uses cached token
- Call setPersistence() — explicitly forbidden per STATE.md decision log
