# Phase 09.1: Page Collections and Index Pattern - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Allow admins to designate pages as Collection indexes and create child Entry pages scoped under them. Eliminates manual slug path typing for structured content (blogs, albums, portfolios). Front-end rendering of collection lists is the developer's responsibility via a new `getCollectionPages` server utility.

Scope:
- New `pageType` field ('collection' | 'page') on page documents ‚Äî set at creation, immutable
- `parentSlug` field on Entry pages that belong to a collection
- `isCollectionIndex` flag on Collection pages
- CreatePageModal updated: type selector + parent picker (collections only) + prefix-locked slug input
- PageManager updated: grouped display (COLLECTIONS section + PAGES section), collections expandable
- New `getCollectionPages(db, parentSlug)` Firestore helper
- Export `getCollectionPages` from `jeeby-cms/server` entry point

Out of scope:
- Nested collections (grandchild pages)
- Reordering entries within a collection (manual sort order)
- Deleting a collection that still has entries (guard required ‚Äî prompt admin to reassign/delete children first)
- Front-end block type for rendering collection index (developer uses `getCollectionPages`)
- Search/filter within a single collection's entries

</domain>

<decisions>
## Implementation Decisions

### Data Model
- D-01: Child (Entry) pages use leaf slug as Firestore document ID ‚Äî no slashes. e.g. doc ID `my-post`, not `blog/my-post`. Slash in doc ID breaks Firestore path parsing.
- D-02: `parentSlug` field (string) on Entry pages points to parent collection's slug. e.g. `parentSlug: 'blog'`. Pages with no parent omit the field (or null).
- D-03: `pageType` field on all new pages: `'collection'` or `'page'`. Existing pages without `pageType` are treated as `'page'` implicitly.
- D-04: Collection pages additionally get `isCollectionIndex: true` for explicit querying.
- D-05: Full URL path for an entry is derived as `/{parentSlug}/{slug}` ‚Äî not stored as a single field. Consumer uses this derivation for routing.
- D-06: Slug uniqueness enforced on the `{parentSlug}/{slug}` pair, not the slug alone. Two collections can each have a child `my-post`.

### Page Types
- D-07: Two types only: "Collection" (index, can have children) and "Page" (leaf, can optionally belong to a collection).
- D-08: Type is selected at creation in CreatePageModal and is immutable ‚Äî no post-creation type change.
- D-09: Only Collection pages appear in the parent picker dropdown.

### CreatePageModal Slug Composition
- D-10: When type is "Page" and a parent collection is selected, the slug input shows the parent prefix as locked read-only text (e.g. `/blog/`) followed by a text input for the leaf segment only.
- D-11: Auto-slug from page name (toKebabSlug) applies to the leaf segment only when a parent is selected.
- D-12: When type is "Collection", no parent picker appears ‚Äî collections are always top-level.
- D-13: Slug uniqueness check on submit uses the full derived path (`parentSlug + '/' + slug`) to detect collisions.

### PageManager Display
- D-14: Grouped layout with two sections: "COLLECTIONS" and "PAGES".
- D-15: Each collection in the COLLECTIONS section is expandable/collapsible ‚Äî default expanded. Shows its child entries indented below.
- D-16: PAGES section shows standard pages (no parent) in the existing flat list with all current sort/filter behavior unchanged.
- D-17: Existing sort/filter controls apply to the top-level items (collection headers + standalone pages). Entries within a collection always sort by `updatedAt` desc (same default as page list).
- D-18: Collection header row shows collection name, status badge, edit button ‚Äî same row shape as existing page rows.

### Rename/Delete Cascade
- D-19: Renaming a collection's slug: all child Entry pages must have their `parentSlug` updated. Non-atomic (same trade-off as existing `renamePage`). Implementation can batch-update children after renaming parent.
- D-20: Deleting a collection that has children: blocked with an inline error ‚Äî "This collection has {N} entries. Delete or reassign them first."

### Front-End Export
- D-21: New `getCollectionPages(db, parentSlug)` function added to `src/firebase/firestore.js`. Queries `where('parentSlug', '==', parentSlug)` ‚Äî returns all entries regardless of draft/published status (same as `listPages`).
- D-22: `getCollectionPages` exported from the `jeeby-cms/server` entry point so Next.js pages can call it server-side.

### Claude's Discretion
- Default expand/collapse state for collections in PageManager (suggest: expanded)
- CSS classes for grouped view headers and entry indent (follow existing jeeby-cms-* naming)
- Whether `getCollectionPages` supports pagination or returns all entries (suggest: all, collections are typically small)
- Exact shape of the type selector UI in CreatePageModal (radio buttons vs segmented control vs select)
- How entries within a collection are ordered in the grouped PageManager view

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ModalShell` ‚Äî CreatePageModal already uses it; no change needed
- `toKebabSlug` in CreatePageModal ‚Äî reuse for leaf slug auto-generation
- `validateSlug(pattern, slug)` ‚Äî still applies to leaf segment against template pattern
- `renamePage(db, oldSlug, newSlug)` ‚Äî pattern to follow for non-atomic collection rename + child update

### Established Patterns
- Slug as Firestore doc ID: preserved for all pages. Child entries use leaf slug only.
- `listPagesPaginated` uses `orderBy('updatedAt', 'desc')` + `startAfter` cursor ‚Äî same pattern for `getCollectionPages` if paginated
- CSS: `jeeby-cms-*` classes, inline styles only for runtime-dynamic values (opacity, z-index)
- Accessibility: WCAG AA enforced; all interactive elements need labels, focus management

### Integration Points
- CreatePageModal: new type selector + conditional parent picker + prefix-locked slug field
- PageManager: grouped rendering replacing flat list; needs `listPages` to return `pageType` and `parentSlug` fields
- Firestore: new `getCollectionPages` query function; `savePage` already uses merge so new fields add transparently
- Server entry point: re-export `getCollectionPages` alongside existing server exports

</code_context>
