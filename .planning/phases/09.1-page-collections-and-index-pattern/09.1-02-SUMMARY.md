## What Was Built

### `getCollectionPages(db, parentSlug)`

Exported from `src/firebase/firestore.js`. Queries the `pages` collection using `where('parentSlug', '==', parentSlug)` combined with `orderBy('updatedAt', 'desc')`. Returns an array of `{ slug, ...data }` objects. Follows the same structural pattern as `listPagesPaginated`. Requires the composite index declared in `firestore.indexes.json`.

### `renameCollection(db, oldSlug, newSlug)`

Exported from `src/firebase/firestore.js`. Non-atomic cascade rename:
1. Fetches children with `getCollectionPages(db, oldSlug)` BEFORE any mutation
2. Renames parent doc via `renamePage(db, oldSlug, newSlug)`
3. Updates each child's `parentSlug` and `updatedAt` via `updateDoc`

The fetch-before-rename ordering guards against the race condition described in RESEARCH Pitfall 4: if children were fetched after the parent rename, any external writer could modify child `parentSlug` values between the delete and the re-fetch, causing non-deterministic results.

### `firestore.indexes.json`

New file at repo root. Firebase CLI format composite index for the `pages` collection:
- `parentSlug` ASCENDING
- `updatedAt` DESCENDING
- `__name__` DESCENDING (standard tiebreaker)

Deploy via `firebase deploy --only firestore:indexes`. Consumers who create the index via Firebase Console (following the auto-generated link on first query) do not need this file.
