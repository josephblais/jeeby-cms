## User Constraints (from CONTEXT.md)

### Locked Decisions

- D-01: Child (Entry) pages use leaf slug as Firestore document ID ‚Äî no slashes. e.g. doc ID `my-post`, not `blog/my-post`. Slash in doc ID breaks Firestore path parsing.
- D-02: `parentSlug` field (string) on Entry pages points to parent collection's slug. e.g. `parentSlug: 'blog'`. Pages with no parent omit the field (or null).
- D-03: `pageType` field on all new pages: `'collection'` or `'page'`. Existing pages without `pageType` are treated as `'page'` implicitly.
- D-04: Collection pages additionally get `isCollectionIndex: true` for explicit querying.
- D-05: Full URL path for an entry is derived as `/{parentSlug}/{slug}` ‚Äî not stored as a single field. Consumer uses this derivation for routing.
- D-06: Slug uniqueness enforced on the `{parentSlug}/{slug}` pair, not the slug alone. Two collections can each have a child `my-post`.
- D-07: Two types only: "Collection" (index, can have children) and "Page" (leaf, can optionally belong to a collection).
- D-08: Type is selected at creation in CreatePageModal and is immutable ‚Äî no post-creation type change.
- D-09: Only Collection pages appear in the parent picker dropdown.
- D-10: When type is "Page" and a parent collection is selected, the slug input shows the parent prefix as locked read-only text (e.g. `/blog/`) followed by a text input for the leaf segment only.
- D-11: Auto-slug from page name (`toKebabSlug`) applies to the leaf segment only when a parent is selected.
- D-12: When type is "Collection", no parent picker appears ‚Äî collections are always top-level.
- D-13: Slug uniqueness check on submit uses the full derived path (`parentSlug + '/' + slug`) to detect collisions.
- D-14: Grouped layout with two sections: "COLLECTIONS" and "PAGES".
- D-15: Each collection in the COLLECTIONS section is expandable/collapsible ‚Äî default expanded. Shows its child entries indented below.
- D-16: PAGES section shows standard pages (no parent) in the existing flat list with all current sort/filter behavior unchanged.
- D-17: Existing sort/filter controls apply to the top-level items (collection headers + standalone pages). Entries within a collection always sort by `updatedAt` desc.
- D-18: Collection header row shows collection name, status badge, edit button ‚Äî same row shape as existing page rows.
- D-19: Renaming a collection's slug: all child Entry pages must have their `parentSlug` updated. Non-atomic. Batch-update children after renaming parent.
- D-20: Deleting a collection that has children: blocked with an inline error ‚Äî "This collection has {N} entries. Delete or reassign them first."
- D-21: `getCollectionPages(db, parentSlug)` added to `src/firebase/firestore.js`. Queries `where('parentSlug', '==', parentSlug)`.
- D-22: `getCollectionPages` exported from the `jeeby-cms/server` entry point.

## Summary

This phase adds a two-tier page hierarchy (Collection + Page entries) to an already-built CMS. The data model is entirely additive: new fields (`pageType`, `parentSlug`, `isCollectionIndex`) are merged into existing Firestore documents with no schema migration required, because `savePage` already uses `{ merge: true }`.

Three files get meaningful edits: `src/firebase/firestore.js` (new `getCollectionPages` function + `where` import), `src/admin/CreatePageModal.js` (type selector + conditional parent picker + split slug input), and `src/admin/PageManager.js` (grouped rendering replacing the flat table loop). One re-export is added to `src/server/index.js`. New CSS classes are added to `styles/cms.css` for the grouped section headers and indented entry rows.

The phase contains no new external dependencies. Every pattern needed ‚Äî Firestore `where` queries, React controlled inputs, Framer Motion presence animation, WCAG focus management ‚Äî is already demonstrated in the codebase.

**Primary recommendation:** Implement in four sequential waves: (1) Firestore layer + tests, (2) CreatePageModal updates + tests, (3) PageManager grouped view + tests, (4) server re-export + build verification.

## Architecture Patterns

### Firestore: getCollectionPages

The new function follows the same structural pattern as `listPages`:

```js
// Source: src/firebase/firestore.js (existing listPages pattern)
export async function getCollectionPages(db, parentSlug) {
  const col = collection(db, 'pages')
  const snap = await getDocs(query(col, where('parentSlug', '==', parentSlug), orderBy('updatedAt', 'desc')))
  return snap.docs.map(d => ({ slug: d.id, ...d.data() }))
}
```

Key notes:
- Firestore `where` + `orderBy` on different fields requires a composite index: `parentSlug ASC, updatedAt DESC`. This index will not exist yet. Firestore will throw with a console link to create it. The planner must include a "create composite index" task or note.
- Returns all entries regardless of draft/published status (per D-21 ‚Äî same behavior as `listPages`).
- No pagination for now (per Claude's Discretion: collections are small).

### Firestore: savePage for new page types

No changes to `savePage`. Because it uses `{ merge: true }`, passing new fields (`pageType`, `parentSlug`, `isCollectionIndex`) in the data object transparently adds them to the document.

Existing pages have no `pageType` ‚Äî per D-03 these are treated as `'page'` implicitly in all UI code. No backfill migration needed.

### Firestore: renameCollection (D-19)

A new function or an extended `renamePage` call sequence:

```js
// Pattern: follow existing renamePage, then patch children
async function renameCollection(db, oldSlug, newSlug) {
  await renamePage(db, oldSlug, newSlug) // copies doc, deletes old
  // Fetch children then batch-update parentSlug
  const children = await getCollectionPages(db, oldSlug)
  await Promise.all(children.map(child =>
    updateDoc(doc(db, 'pages', child.slug), { parentSlug: newSlug })
  ))
}
```

This is non-atomic (same accepted trade-off as existing `renamePage`). Children must be fetched before the parent is renamed (since `getCollectionPages` queries by `parentSlug`). Correct order: fetch children first, rename parent, then update children.

Corrected ordering:

```js
async function renameCollection(db, oldSlug, newSlug) {
  const children = await getCollectionPages(db, oldSlug) // fetch while old slug exists
  await renamePage(db, oldSlug, newSlug)
  await Promise.all(children.map(child =>
    updateDoc(doc(db, 'pages', child.slug), { parentSlug: newSlug })
  ))
}
```

### Firestore: delete collection guard (D-20)

Before calling `deletePage`, check child count:

```js
const children = await getCollectionPages(db, slug)
if (children.length > 0) {
  // surface inline error: "This collection has N entries. Delete or reassign them first."
  return
}
await deletePage(db, slug)
```

This check lives in PageManager, not in `deletePage` itself (consistent with existing pattern where DeletePageModal calls `deletePage` directly).

### CreatePageModal: type selector

A `<select>` element is simplest and maximally accessible without custom ARIA. Segmented controls require `role="radiogroup"` + keyboard roving ‚Äî higher complexity for no UX gain here. Use a labeled `<select>`:

```jsx
<div className="jeeby-cms-field">
  <label htmlFor="cms-page-type">Page type</label>
  <select id="cms-page-type" value={pageType} onChange={e => setPageType(e.target.value)}>
    <option value="page">Page</option>
    <option value="collection">Collection</option>
  </select>
</div>
```

When `pageType === 'collection'`: hide parent picker, slug input reverts to plain input (no prefix).
When `pageType === 'page'`: show optional parent picker.

### CreatePageModal: parent picker

A `<select>` populated from collections in the existing `listPages` result (filtered to `pageType === 'collection'`). The modal already calls `listPages` for slug uniqueness checking ‚Äî reuse that fetch to populate the picker. Filter on client side.

```jsx
const collections = existingPages.filter(p => p.pageType === 'collection')
// Render:
<select id="cms-parent-collection" value={parentSlug} onChange={...}>
  <option value="">None (top-level page)</option>
  {collections.map(c => <option key={c.slug} value={c.slug}>/{c.slug}</option>)}
</select>
```

The `listPages` call currently happens only on submit. For the parent picker to be populated before submit, move the `listPages` fetch to occur when the modal opens (or lazily on first interaction with the parent picker). Simplest: fetch on open if `pageType === 'page'`, cache result in state.

### CreatePageModal: split slug input (D-10)

When `pageType === 'page'` and `parentSlug` is set, show a prefix + leaf input pair:

```jsx
{parentSlug ? (
  <div className="jeeby-cms-slug-prefixed">
    <span className="jeeby-cms-slug-prefix" aria-hidden="true">/{parentSlug}/</span>
    <input
      id="cms-page-slug"
      type="text"
      value={slug}
      aria-label="Slug (leaf segment)"
      aria-describedby="cms-slug-hint cms-slug-error"
      onChange={...}
    />
  </div>
) : (
  <input id="cms-page-slug" type="text" value={slug} ... />
)}
<p id="cms-slug-hint">
  {parentSlug ? `Full path: /${parentSlug}/${slug || 'my-slug'}` : 'e.g. /about or /contact'}
</p>
```

The hidden prefix text must NOT be part of the input value ‚Äî the leaf segment alone is stored as the Firestore doc ID (D-01). The hint text showing the derived full path is helpful for editors to understand what URL they're creating.

Slug uniqueness check on submit (D-13): compare `parentSlug + '/' + slug` against the full derived paths of existing pages:

```js
const fullPath = parentSlug ? `${parentSlug}/${slug}` : slug
const clash = existing.some(p => {
  const existingPath = p.parentSlug ? `${p.parentSlug}/${p.slug}` : p.slug
  return existingPath === fullPath
})
```

### PageManager: grouped display

The existing rendering is a single flat `<table>`. The grouped view should keep the `<table>` but use `<tbody>` sections with visually distinct section header rows (not `<th scope="rowgroup">` ‚Äî that's for column grouping, not visual sections).

Pattern: use separate `<tbody>` elements per section, or a single `<tbody>` with explicit "section header" rows that span all columns. The single-`<tbody>` approach is simpler and avoids keyboard focus issues with multiple `<tbody>` elements.

```jsx
<tbody>
  {/* COLLECTIONS section header */}
  <tr className="jeeby-cms-table-section-header">
    <td colSpan={5}>Collections</td>
  </tr>
  {collections.map(coll => (
    <Fragment key={coll.slug}>
      {/* Collection header row ‚Äî expandable */}
      <tr className="jeeby-cms-collection-row">
        <td>
          <button
            type="button"
            aria-expanded={expandedCollections.has(coll.slug)}
            aria-controls={`cms-collection-entries-${coll.slug}`}
            className="jeeby-cms-collection-toggle"
            onClick={() => toggleCollection(coll.slug)}
          >
            <IconChevron expanded={expandedCollections.has(coll.slug)} />
            {coll.name || coll.slug}
          </button>
        </td>
        {/* ... status, actions columns */}
      </tr>
      {/* Entry rows ‚Äî shown/hidden via aria-hidden + CSS */}
      {expandedCollections.has(coll.slug) && entries
        .filter(e => e.parentSlug === coll.slug)
        .map(entry => (
          <tr key={entry.slug} className="jeeby-cms-entry-row" id={`cms-collection-entries-${coll.slug}`}>
            <td className="jeeby-cms-entry-indent">
              {/* entry name/slug/status/actions */}
            </td>
          </tr>
        ))
      }
    </Fragment>
  ))}
  {/* PAGES section header */}
  <tr className="jeeby-cms-table-section-header">
    <td colSpan={5}>Pages</td>
  </tr>
  {standalonePages.map(page => (
    /* existing row rendering unchanged */
  ))}
</tbody>
```

ARIA note: `aria-controls` on the toggle button pointing to the entry rows' container gives AT users a way to jump to the controlled region. `aria-expanded` communicates collapse state. This follows the WAI-ARIA disclosure pattern.

### PageManager: data split

`listPages` returns all pages in one array. Split client-side:

```js
const collections = pages.filter(p => p.pageType === 'collection')
const entries     = pages.filter(p => p.parentSlug)
const standalone  = pages.filter(p => !p.pageType && !p.parentSlug || p.pageType === 'page' && !p.parentSlug)
```

For backwards compatibility, pages without `pageType` (created before this phase) are treated as standalone `'page'` items (D-03).

Sort/filter per D-17: apply `applySortFilter` and `applySearch` to the combined top-level items array (collections + standalone pages). Entries inside a collection are always sorted `updatedAt` desc, independently of the active sort mode.

### Server entry point re-export (D-22)

`src/server/index.js` currently imports from `../firebase/admin.js` (Firebase Admin SDK). `getCollectionPages` uses the client SDK. It cannot be called from `src/server/index.js` directly ‚Äî the function signature takes a `db` from `firebase/firestore` (client), not `firebase-admin`.

Two approaches:
1. Re-export `getCollectionPages` from `src/server/index.js` with a comment that the caller must supply an Admin Firestore instance (using `firebase-admin`'s `.collection()` API which is compatible with the same `where`/`query` pattern).
2. Export `getCollectionPages` from both `src/server/index.js` AND from `jeeby-cms` (client-side), with documentation for which to use where.

The simpler correct approach: export `getCollectionPages` as-is from `src/server/index.js`. The consumer will pass an Admin Firestore `db` instance (from `getAdminFirestore()`). Since `getCollectionPages` only calls `collection()`, `getDocs()`, `query()`, `where()`, `orderBy()` ‚Äî these are identical in the Admin SDK (via `firebase-admin/firestore`). However, the current `getCollectionPages` imports from `'firebase/firestore'` (client SDK). This creates a module mismatch when called server-side.

Recommended resolution: `src/server/index.js` re-exports from a server-safe wrapper that uses `firebase-admin/firestore`. OR: document that `getCollectionPages` is a client-side utility and export it from the main `jeeby-cms` entry only (not `jeeby-cms/server`). This contradicts D-22.

Correct interpretation of D-22: The consumer calls `getCollectionPages` in a Next.js Server Component (e.g., `app/blog/page.js`). They already have an Admin Firestore `db` from `getAdminFirestore()`. The function needs to work with the Admin SDK.

Resolution the planner should implement: Create `getCollectionPages` using Admin SDK-compatible calls in `src/server/index.js` (or a shared helper that accepts any Firestore-like `db`), OR make `getCollectionPages` accept a generic `db` and rely on duck-typing (Admin and Client SDKs have identical `collection()`/`query()`/`where()`/`getDocs()` surfaces for basic reads). The existing `getCMSContent` uses Admin SDK's `db.doc('pages/'+slug).get()` ‚Äî the Admin SDK has a different API shape for collection queries (`db.collection('pages').where(...)` vs the modular client SDK). These are NOT compatible without a wrapper.

ACTION FOR PLANNER: Add a task to investigate and decide: either (a) use the Admin SDK's modular API (`firebase-admin/firestore`'s `getFirestore`, `collection`, `query`, `where`, `getDocs`), or (b) implement `getCollectionPages` with the Admin SDK's legacy chained API (`db.collection('pages').where(...).get()`). Option (b) is consistent with how `getCMSContent` currently works (`db.doc('pages/'+slug).get()`). Recommend option (b) for consistency with existing server code.

### Anti-Patterns to Avoid

- Do not store the full URL path (`/blog/my-post`) as a Firestore field ‚Äî D-05 explicitly forbids this. Derive it at render time.
- Do not put slashes in Firestore document IDs ‚Äî D-01. Firestore treats `/` as a path separator; a doc ID like `blog/my-post` would silently create a subcollection path.
- Do not change type after creation ‚Äî D-08. No UI affordance or API should allow this.
- Do not apply the active sort/filter to entries within a collection ‚Äî D-17. Entry order is always `updatedAt` desc.
- Do not use `<div role="button">` for the collection expand toggle ‚Äî use `<button>`.
- Do not use `aria-controls` with an ID that maps to multiple rows ‚Äî point it to a container wrapper.

## Common Pitfalls

### Pitfall 1: Firestore composite index missing for getCollectionPages

**What goes wrong:** `getCollectionPages` uses `where('parentSlug', '==', parentSlug)` combined with `orderBy('updatedAt', 'desc')`. Firestore requires a composite index for `where` on one field + `orderBy` on a different field. The query will throw a FirebaseError with a console URL to create the index.

**Why it happens:** Firestore auto-creates single-field indexes but not composite ones.

**How to avoid:** Either (a) include the index definition in `firestore.indexes.json` if the project uses Firebase CLI, or (b) document in the plan that the developer must create the index after deployment. In development, clicking the console URL is sufficient.

**Warning signs:** `FirebaseError: The query requires an index` in the browser console after implementing `getCollectionPages`.

### Pitfall 2: listPages returns everything ‚Äî entries appear in the PAGES section

**What goes wrong:** `listPages` returns all pages including entries (child pages with `parentSlug`). If the split logic doesn't filter out entries, they will appear in both the COLLECTIONS children rows AND the PAGES section.

**Why it happens:** Existing `listPages` has no awareness of page hierarchy.

**How to avoid:** The standalone pages filter must explicitly exclude entries: `p.pageType !== 'collection' && !p.parentSlug`.

### Pitfall 3: Slug uniqueness check misses cross-collection collisions

**What goes wrong:** Current uniqueness check is `existing.some(p => p.slug === slug)`. After this phase, two collections can each have `my-post`. The check must compare the full derived path.

**Why it happens:** The check was written before parent slugs existed.

**How to avoid:** Replace with `existing.some(p => (p.parentSlug ? p.parentSlug + '/' + p.slug : p.slug) === fullPath)` where `fullPath = parentSlug ? parentSlug + '/' + slug : slug`.

### Pitfall 4: renameCollection fetches children after renaming parent

**What goes wrong:** If `renamePage` is called first (deletes old doc), then `getCollectionPages(db, oldSlug)` returns zero results because there are no pages with `parentSlug === oldSlug` any more. Children are orphaned with a stale `parentSlug`.

**Why it happens:** Wrong operation order in rename sequence.

**How to avoid:** Always fetch children BEFORE renaming the parent (see Architecture Patterns above).

### Pitfall 5: CreatePageModal listPages called only on submit

**What goes wrong:** The parent picker dropdown is empty until the user submits because `listPages` is only called on submit in the current code.

**Why it happens:** Current modal only needs pages list for uniqueness check at submit time.

**How to avoid:** Trigger `listPages` when the modal opens (in the `useEffect([open])` reset block). Cache result in component state. Reuse the same cached result for both the parent picker population and the uniqueness check on submit.

### Pitfall 6: aria-controls pointing to multiple tr elements

**What goes wrong:** `aria-controls` on the collection toggle button should reference a single region ID. If multiple `<tr>` rows have the same `id`, HTML is invalid and AT behavior is undefined.

**Why it happens:** Lazy ID assignment in a list render.

**How to avoid:** Wrap all entry rows for a collection in a single container. In a table context, use a `<tbody id="cms-collection-entries-{slug}">` for each collection's entries. Multiple `<tbody>` elements in one `<table>` is valid HTML.

## Code Examples

### getCollectionPages (client SDK flavor)

```js
// Source: follows listPagesPaginated pattern in src/firebase/firestore.js
export async function getCollectionPages(db, parentSlug) {
  const col = collection(db, 'pages')
  const snap = await getDocs(
    query(col, where('parentSlug', '==', parentSlug), orderBy('updatedAt', 'desc'))
  )
  return snap.docs.map(d => ({ slug: d.id, ...d.data() }))
}
```

### getCollectionPages (Admin SDK flavor for server/index.js)

```js
// Source: follows getCMSContent pattern in src/server/index.js
export async function getCollectionPages(parentSlug) {
  const db = getAdminFirestore()
  const snap = await db.collection('pages')
    .where('parentSlug', '==', parentSlug)
    .orderBy('updatedAt', 'desc')
    .get()
  return snap.docs.map(d => ({ slug: d.id, ...d.data() }))
}
```

Note: Admin SDK version does not take `db` as parameter ‚Äî it calls `getAdminFirestore()` internally, consistent with how `getCMSContent` works.

### Collection expand toggle state

```js
// Default all expanded (Claude's Discretion)
const [expandedCollections, setExpandedCollections] = useState(() =>
  new Set(pages.filter(p => p.pageType === 'collection').map(p => p.slug))
)

function toggleCollection(slug) {
  setExpandedCollections(prev => {
    const next = new Set(prev)
    next.has(slug) ? next.delete(slug) : next.add(slug)
    return next
  })
}
```

### Accessibility: collection toggle button


The chevron icon should rotate visually (CSS transform) when expanded. Use `data-expanded` attribute on the button for the CSS hook ‚Äî consistent with how `data-filter-active` is used on `.jeeby-cms-sort-trigger`.

## Open Questions

1. **Admin SDK vs Client SDK for server `getCollectionPages`**
   - What we know: `getCMSContent` uses Admin SDK chained API (`db.doc(...).get()`). D-22 says export from `jeeby-cms/server`. Client SDK `getCollectionPages` uses modular API (`query`, `where`, `getDocs`).
   - What's unclear: Does the planner want a single function used both client and server-side, or separate implementations?
   - Recommendation: Implement two versions ‚Äî one in `src/firebase/firestore.js` (client SDK, for admin UI internal use if needed), one in `src/server/index.js` (Admin SDK chained API, for consumer Server Components). The server version does not take `db` as a parameter (same as `getCMSContent`).

2. **Firestore composite index**
   - What we know: Required for `where('parentSlug') + orderBy('updatedAt')`.
   - What's unclear: Does the project use Firebase CLI with `firestore.indexes.json`? No such file was found in the repo.
   - Recommendation: Include a task to create the composite index manually via Firebase Console, and document it in a README or code comment. Flag this as a deployment step.

3. **aria-controls for multi-row entry section**
   - What we know: Standard ARIA pattern uses `aria-controls` pointing to a single element ID.
   - What's unclear: Best HTML structure in a `<table>` context for a collapsible group.
   - Recommendation: Use multiple `<tbody>` elements ‚Äî one per collection's entries ‚Äî each with a unique `id`. Multiple `<tbody>` in one `<table>` is valid HTML5. This gives `aria-controls` a clean single target per collection.
