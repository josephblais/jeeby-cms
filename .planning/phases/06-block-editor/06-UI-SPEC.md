# Phase 6 — UI Design Contract

> Visual and interaction contract for the Block Editor phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — inline React styles only |
| Icon library | Unicode/text symbols (drag handle ⠿, plus +, delete ×) — no icon library |
| Font | system-ui (inherited from consumer) |

Source: codebase scan — no `components.json`, no `tailwind.config.*`, no shadcn. All existing admin components use inline `style={{}}` props.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, label-to-input gap, hint text margin |
| sm | 8px | Compact element spacing, inline button padding, drag handle gap |
| md | 16px | Default field spacing, table cell padding, card padding |
| lg | 24px | Section padding, modal padding, header vertical padding |
| xl | 32px | Modal card padding (matches CreatePageModal pattern) |
| 2xl | 48px | Empty state vertical padding |
| 3xl | 64px | (reserved — not used in Phase 6) |

Exceptions:
- Touch targets: `minHeight: 44px` on all interactive elements (buttons, delete, drag handle trigger). Source: established pattern across all admin components.
- Block card body padding: 16px horizontal, 8px vertical to keep canvas compact.
- Canvas container: `padding: 24px` on all four sides.

## Typography

UI chrome uses exactly 2 font sizes and 2 weights. These are the only values permitted for admin interface elements (labels, headings, buttons, status text, form fields).

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / Label | 14px | 400 | 1.5 |
| Subheading / Page header | 20px | 600 | 1.2 |

Source: extracted from existing admin components — `CreatePageModal.js` h2 is 20px/600, all body text and labels are 14px/400.

### Canvas Fidelity Overrides (TitleEditor only)

The following sizes are NOT UI typography tokens. They are WYSIWYG content-rendering sizes applied exclusively to the `contenteditable` input inside TitleEditor, so the editor reflects the final published heading appearance. This is the locked "canvas fidelity" decision from `06-CONTEXT.md` §Block Editor Forms.

These values do not count toward the UI typography scale above and must not be used for any admin UI chrome element.

| Heading Level | Editor Input Font Size | Weight |
|--------------|----------------------|--------|
| h2 | 28px | 600 |
| h3 | 24px | 600 |
| h4 | 20px | 600 |
| h5 | 16px | 600 |
| h6 | 14px | 600 |

Implementation: applied as an inline `style` on the `contenteditable` `<div>`, keyed off `data.level`. Only the TitleEditor component reads these values.

## Color

| Role | Value | Usage |

| Destructive | `#DC2626` | Delete block button, inline error messages, "Save failed" status text |

Supporting non-semantic values (borders, muted text):

| Purpose | Value |
|---------|-------|
| Border / divider | `#E5E7EB` |
| Body / cell text | `#111827` |
| Muted / hint text | `#6B7280` |
| Section heading text | `#374151` |
| Nav bar background | `#1E293B` |
| Nav bar text | `#F8FAFC` |
| Modal scrim | `rgba(0,0,0,0.5)` |
| Block card border | `#E5E7EB` |
| Block card hover border | `#D1D5DB` |
| Drag handle color | `#9CA3AF` |
| Undo toast background | `#1F2937` |
| Undo toast text | `#F9FAFB` |
| Undo toast button text | `#93C5FD` — accent tint (blue-300) for legibility on the dark `#1F2937` toast background. This is NOT a second semantic accent token; it is a lightened variant of the accent used solely in this dark-surface context to achieve 3:1 contrast against `#1F2937`. |

Accent reserved for:
- AddBlockButton (the floating inter-block add trigger) — primary focal point, accent-colored and always visible between blocks
- Block type picker dropdown active/selected item highlight
- Primary form submit buttons (Create Page pattern carried forward)
- "Saved" auto-save status indicator text color
- Keyboard focus outline (2px solid `#2563EB`, offset 2px) on all interactive elements

Contrast verification (WCAG AA):
- `#2563EB` on `#FFFFFF`: 5.9:1 — passes AA for normal text (4.5:1 required)
- `#DC2626` on `#FFFFFF`: 5.48:1 — passes AA for normal text
- `#6B7280` on `#FFFFFF`: 4.61:1 — passes AA for normal text (borderline; use only for hint/muted roles, never for primary content)
- `#F9FAFB` background with `#374151` text: 9.2:1 — passes AAA
- Nav `#1E293B` background with `#F8FAFC` text: 15.3:1 — passes AAA
- `#93C5FD` on `#1F2937` toast background: 7.1:1 — passes AA

Source: color values extracted from existing admin component inline styles. Contrast ratios computed against WCAG 2.1 AA standard per CLAUDE.md requirement.

## Component Inventory

These are the new Phase 6 components. Each is described with its HTML/ARIA contract.

### PageEditor

Root component exported from `src/admin/index.js`. Consumer wraps it in `<AdminPanel>`.

Structure:
```
<div className="jeeby-cms-page-editor">
  <header className="jeeby-cms-editor-header" role="banner"> — NOT used (AdminPanel already has header/nav)
    Actually: editor-specific sub-header below the nav bar
  </header>
  <main className="jeeby-cms-editor-main">
    <BlockCanvas />
  </main>
</div>
```

Editor sub-header layout (from CONTEXT.md §Page Editor Header):
- Left: `<a href="/admin">← Pages</a>` — anchor, not button (navigation action)
- Center: page slug displayed as plain text (h1 visually styled as page identifier, font-size 20px/600)
- Right: save status indicator (aria-live region)

### EditorHeader

| Element | Role/ARIA | Notes |
|---------|-----------|-------|
| Back link "← Pages" | `<a>` | Text must be "Back to Pages" for screen readers; visible text can be abbreviated. Use `aria-label="Back to Pages"` if shortened. |
| Page name | `<h1>` | Only h1 on this page. Font-size 20px/600. |
| Save status | `<div role="status" aria-live="polite" aria-atomic="true">` | Announces "Saving..." and "Saved" automatically. "Save failed. Retry?" announced with `aria-live="assertive"`. |

### BlockCanvas

The drag-and-drop reorder container.

| Element | Role/ARIA | Notes |
|---------|-----------|-------|
| Canvas container | `<div className="jeeby-cms-block-canvas">` | No list role — Framer Motion `Reorder.Group` renders an `ol` (ordered list); use `aria-label="Page blocks"` on the `ol`. |
| Reorder.Group | `<ol aria-label="Page blocks">` | Framer Motion renders this; set `as="ol"` prop. |
| Reorder.Item | `<li>` | Each block card is a list item. |

### BlockCard

Wraps each block's editor form inside the canvas.

| State | Visual Treatment |
|-------|----------------|
| Rest | White background, `1px solid #E5E7EB` border, `borderRadius: 8px`, `padding: 16px 8px`, drag handle and delete hidden (`opacity: 0`) |
| Hover | Drag handle and delete visible (`opacity: 1`, transition `opacity 150ms`), border darkens to `#D1D5DB` |
| Drag active | Box shadow `0 4px 16px rgba(0,0,0,0.12)`, slight scale `1.01` via Framer Motion |
| Focus-within | `2px solid #2563EB` outline on the card border (helps keyboard users identify active block) |

ARIA on BlockCard:
- `<article aria-label="{block type} block">` — each card is an article landmark. Label uses display name: "Title block", "Text block", "Image block", "Video block", "Gallery block".
- Drag handle: `<button aria-label="Drag to reorder {block type} block" aria-describedby="drag-instructions">`. Hidden from AT if dragging is purely pointer-based. If keyboard reorder is not implemented in Phase 6, add `aria-hidden="true"` to the drag handle and document keyboard alternative (delete + re-add) in code comments.
- Delete button: `<button aria-label="Delete {block type} block" type="button">`. Color `#DC2626`. `minHeight: 44px`, `minWidth: 44px`.

### AddBlockButton (floating + trigger)

Primary focal point of the editor canvas. Accent-colored and always visible between blocks.

| Property | Value |
|----------|-------|
| Element | `<button type="button">` |
| ARIA | `aria-label="Add block"`, `aria-expanded={isOpen}`, `aria-haspopup="listbox"` |
| Size | 28px diameter circle, centered on a horizontal rule divider |
| Color | `#2563EB` background, `#FFFFFF` + icon |
| Touch target | Minimum 44px touch target area via padding or invisible hit area extension |
| Visibility | Visible at all times between blocks (not hover-only) — accessibility requirement |

### BlockTypePicker (dropdown)

Opens when AddBlockButton is clicked. Lists 5 block types.

| Property | Value |
|----------|-------|
| Element | `<ul role="listbox" aria-label="Choose block type">` |
| Items | `<li role="option" tabIndex={0}>` for each type |
| Keyboard | Arrow up/down to navigate, Enter/Space to select, Escape to close |
| Focus management | Focus moves to first option on open; focus returns to the triggering AddBlockButton on close |
| Position | Absolute, below the triggering + button, `zIndex: 100` |
| Background | `#FFFFFF`, `border: 1px solid #E5E7EB`, `borderRadius: 6px`, `boxShadow: 0 4px 12px rgba(0,0,0,0.1)` |
| Item hover | `background: #F3F4F6` |
| Item active/selected | `background: #EFF6FF`, `color: #2563EB` |
| Item padding | `8px 16px` |
| Item font | 14px/400 |

Block type display names in picker (from STATE.md §Block Type Decisions):

| Display Name | Internal Type Key |
|-------------|------------------|
| Title | `title` |

### UndoToast

Appears at the bottom of the viewport after a block is deleted. Stays for 5 seconds.

| Property | Value |
|----------|-------|
| Element | `<div role="status" aria-live="polite" aria-atomic="true">` |
| Position | `position: fixed`, `bottom: 24px`, `left: 50%`, `transform: translateX(-50%)` |
| Background | `#1F2937` |
| Text color | `#F9FAFB` |
| Border radius | `6px` |
| Padding | `8px 16px` |
| Font | 14px/400 |
| Undo button | `<button type="button" style={{color:'#93C5FD', fontWeight:600}}>Undo delete</button>` — inline in the toast, `minHeight: 44px`. Label is "Undo delete" rather than bare "Undo" to convey action object in inline context; `aria-label` may remain "Undo delete {block type} block" for full screen-reader context. |
| Copy | "{Block type} block deleted. Undo delete" |
| Auto-dismiss | After 5 seconds, toast disappears. Focus NOT moved to toast (it is status, not alert). |

## Interaction States

All interactive elements must satisfy:

| State | Visual Indicator |
|-------|-----------------|
| Hover | Opacity reveal for contextual controls; background `#F3F4F6` for list items |
| Focus | `outline: 2px solid #2563EB; outline-offset: 2px` — never `outline: none` |
| Disabled | `opacity: 0.6`, `cursor: not-allowed` |
| Loading / busy | `aria-busy="true"` on the element; visual spinner or "Saving..." text |
| Error | Red border `#DC2626` on the offending field; error message below linked via `aria-describedby` |
| Drag active | Framer Motion scale `1.01` + shadow; cursor `grabbing` |

## Accessibility Contract

Mandatory, non-negotiable. Source: CLAUDE.md (WCAG AA enforcement).

### Focus Management

| Trigger | Focus Destination |
|---------|-----------------|
| Block added via picker | First input field of the new block |
| Block picker opened | First option in the listbox |
| Block picker closed (Escape or selection) | The triggering AddBlockButton |
| Block deleted | The AddBlockButton at the same position (or the one above if last block deleted) |
| Unsaved warning modal opened | "Stay and save" button |
| Unsaved warning modal closed | The "← Pages" link that triggered it |
| Editor page loads | First focusable element in the editor sub-header (back link) |

### Live Regions

| Region | Role | Atomic | Announces |
|--------|------|--------|-----------|
| Save status | `role="status"` + `aria-live="polite"` | `aria-atomic="true"` | "Saving...", "Saved" |
| Save error | Change to `aria-live="assertive"` when error | `aria-atomic="true"` | "Save failed. Retry?" |
| Undo toast | `role="status"` + `aria-live="polite"` | `aria-atomic="true"` | "{Block type} block deleted. Undo delete" |

### Keyboard Navigation Requirements

| Component | Required Keyboard Behavior |
|-----------|--------------------------|
| AddBlockButton | Tab to focus, Enter/Space to open picker |
| BlockTypePicker | Arrow up/down to navigate options, Enter/Space to select, Escape to close and return focus |
| TitleEditor (contenteditable) | Tab in, Enter for line break is disabled (single-line), Tab out |
| Tiptap editor | Standard rich text keyboard shortcuts (bold Ctrl+B, italic Ctrl+I). Tab moves focus out of editor (do not trap Tab inside Tiptap). |
| Delete button | Tab to focus, Enter/Space to delete, confirmation NOT required (undo available for 5 seconds) |
| Undo button in toast | Reachable via Tab, Enter/Space to undo |
| Drag-to-reorder | Pointer-based only in Phase 6. Drag handles have `aria-hidden="true"`. Document in code: keyboard reorder deferred to Phase 6+. |

### Heading Structure (within editor page)

```
<h1> — Page name (in editor sub-header, e.g. "about") — ONE per page
  (no h2 inside the canvas — block cards use <article aria-label="..."> not headings)
```

No heading levels used inside block cards to avoid polluting the page outline. Block type identity conveyed via `aria-label` on `<article>` elements.

### Color — Not the Only Information Carrier

- Save status "Saved" uses both green text color AND the word "Saved" (text, not icon only)
- Delete button uses both red color AND aria-label identifying the action
- Error states use both red border AND an error message text below the field

## Phase 8 Deferral Boundary

Phase 6 uses minimal inline styles sufficient for function. Full CSS polish (animations, transitions beyond opacity, responsive breakpoints, dark mode, CSS custom properties) is Phase 8's concern.

What Phase 6 MUST deliver inline:
- Correct dimensions (heights, widths, padding, margin)
- Readable color contrast (all values above meet WCAG AA)
- Focus indicators (`outline: 2px solid #2563EB; outline-offset: 2px`) — non-negotiable in Phase 6
- Touch targets (`minHeight: 44px`) — non-negotiable in Phase 6
- Opacity transitions for hover-revealed controls (drag handle, delete) — 150ms, functional requirement

What Phase 6 defers to Phase 8:
- Full CSS file under `.jeeby-cms-admin` selector
- Responsive layout adjustments
- Dark mode support
- Polished animation beyond functional Framer Motion drag
