# Phase 09.2: Media Library - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Build a persistent media library backed by a `/cms/media` Firestore collection. Every image uploaded to the CMS — whether from the media library modal or directly from a block editor — gets a library entry with title, alt text, and storage metadata.

Surfaces in three places:
1. **Media Library modal** — opened from ImageEditor ("Select from Library"), GalleryEditor ("Select from Library"), and the home admin page ("Upload Media" button in PageManager)
2. **ImageEditor** — existing Upload button now also creates a library entry + prompts for title/alt inline; new "Select from Library" button
3. **GalleryEditor** — same as ImageEditor, plus multi-select checkout flow when using the library

Scope:
- `/cms/media` Firestore collection with CRUD helpers
- `MediaLibraryModal` component using `ModalShell`
- Lazy-loaded infinite scroll grid (batches ordered by `uploadedAt` desc)
- Upload flow: file picker → progress → inline title/alt form → Save to Library
- Edit flow: click any library image → edit title/alt inline → save
- Block editor integration: "Select from Library" button in ImageEditor and GalleryEditor
- Block editor upload integration: existing Upload buttons now write a library record + show inline title/alt prompt after upload
- Home page integration: "Upload Media" button in PageManager opens the library modal in browse+upload mode

Out of scope:
- Storage file deletion from the library UI (consumer manages lifecycle — Phase 9 decision)
- Video/audio in the media library (image-only for v1)
- Bulk delete or library management features
- Filtering or search within the library (post-v1)

</domain>

<decisions>
## Implementation Decisions

### Firestore Schema
- **D-01:** Media metadata lives in a `/cms/media` Firestore collection — separate from `/cms/pages`, pageable, no 1MB ceiling risk
- **D-02:** Each document contains: `storageUrl` (string), `storagePath` (string — for future delete), `title` (string), `alt` (string), `uploadedAt` (Timestamp), `mimeType` (string), `size` (number, bytes)
- **D-03:** Document ID generated at write time (same `crypto.randomUUID()` pattern used elsewhere)
- **D-04:** Query order: `orderBy('uploadedAt', 'desc')` — most recent first; use cursor pagination (same pattern as `listPagesPaginated`)

### Upload + Metadata Capture Flow
- **D-05:** When uploading from the media library modal — after upload completes, the image card flips inline to show a title + alt text form. User fills in details and clicks "Save to Library" to commit the Firestore document. The card then shows the saved image in the grid.
- **D-06:** Alt text is optional at save time, but a visible warning appears ("Images without alt text may fail accessibility checks") when alt is left blank. Title is optional with no warning.
- **D-07:** When uploading directly from ImageEditor or GalleryEditor — after upload completes, the same inline title/alt prompt appears within the block editor before the URL is filled into the block. User fills in (or skips with warning) and confirms. The library record is created and the URL auto-fills the block field.
- **D-08:** Multi-upload from the media library modal (matching GalleryEditor's batch behavior) — each uploaded file gets its own card flip + title/alt form. Uploads run in parallel; metadata forms appear as each completes.

### Auto-Populate on Selection
- **D-09:** When a library image is selected into a block that has an existing non-empty alt text: a small inline conflict prompt appears inside the block editor after the modal closes — shows the library alt text and offers [Keep mine] / [Replace] buttons
- **D-10:** When the block's alt field is empty: library alt text fills it automatically, no prompt
- **D-11:** Library `title` is not mapped to any block field — it's a library-level label only (used for display in the grid, not surfaced in the front-end block)

### Media Library Modal
- **D-12:** Large centered card: ~90vw / 85vh, backdrop visible behind it. Uses `ModalShell` for focus trap, scroll lock, Escape-to-close, and Framer Motion animation — consistent with existing modals
- **D-13:** Modal has two modes: **browse+upload** (opened from home page — no selection callback, no "Select" action) and **select** (opened from a block editor — images are selectable)
- **D-14:** In select mode, ImageEditor opens the modal in single-select mode (click image → modal closes → block fills). GalleryEditor opens in multi-select mode.
- **D-15:** Infinite scroll: `IntersectionObserver` watches a sentinel element at the bottom of the grid; when visible, fetch the next batch via `listMediaPaginated`. Batch size: Claude's discretion (likely 20–24 items).
- **D-16:** Upload button inside the modal: opens a file picker accepting `image/jpeg,image/png,image/gif,image/webp` (same MIME allowlist as Phase 9). Multiple file selection enabled.

### GalleryEditor Multi-Select
- **D-17:** In multi-select mode, each grid image shows a visible checkbox on hover/focus. Checked images are highlighted. A sticky footer shows "N selected" and an "Add N images" confirm button. Clicking "Add N images" appends all selected items to the gallery and closes the modal.

### Editing Existing Library Items
- **D-18:** Clicking any saved image in the library grid opens an inline edit panel (card flip or expanded card) with editable title and alt fields and a Save button. Changes write back to the `/cms/media` Firestore document. No editing from within block editors.

### Home Page Integration
- **D-19:** "Upload Media" button added to PageManager (the default home admin view). Opens the MediaLibraryModal in browse+upload mode (no selection callback).

### Claude's Discretion
- Exact grid column count and thumbnail aspect ratio
- Infinite scroll batch size (suggested: 20–24)
- Loading skeleton vs spinner while first batch loads
- Card flip animation style for the metadata form
- Exact CSS class naming for new media library elements
- Error state display when Firestore fetch fails
- Empty state design when no media has been uploaded yet

</decisions>

- `ModalShell` — complete modal shell; use for `MediaLibraryModal`; handles focus trap, scroll lock, Escape, Framer Motion animation

- Inline progress feedback (no toasts) — established in Phase 9; keep for upload progress in the library
- CSS classes only, no inline styles except runtime-dynamic values — established in Phase 8
- `jeeby-cms-btn-ghost` for secondary/icon buttons
- WCAG AA enforced throughout; `role="alert"` for status messages, visible labels, focus management on modal open/close

<specifics>
## Specific Ideas

- Modal grid: card-flip animation when upload completes and title/alt form appears — subtle, matches Framer Motion already in the project
- Conflict prompt in block editor: small inline banner below the URL input, not a separate modal — "Library alt: '[value]'" with [Keep mine] / [Replace] buttons; dismisses after choice
- GalleryEditor multi-select sticky footer: fixed to the bottom of the modal card, always visible once at least one item is checked
- "Upload Media" button placement in PageManager: near the top of the page list, alongside or near existing "New Page" action

</specifics>

<deferred>
## Deferred Ideas

- Video/audio support in the media library — out of scope for v1 (image-only)
- Search or filter within the library — post-v1
- Bulk delete or storage cleanup from the library UI — consumer manages Storage lifecycle
- Two-way sync: editing alt text in a block editor updating the library record — post-v1 complexity
- Upload from URL directly into the library (paste-to-library) — post-v1
- Usage tracking (which pages use a given media item) — post-v1

</deferred>
