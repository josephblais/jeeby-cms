# Phase 09.2: Media Library - Research

**Researched:** 2026-03-29
**Domain:** Firestore collection + React modal grid + block editor integration
**Confidence:** HIGH

## Summary

Phase 09.2 adds a persistent media library backed by a `/cms/media` Firestore collection. The
codebase already has all the foundation needed: `uploadFile` + `validateImageFile` in `storage.js`,
cursor-paginated Firestore queries in `firestore.js`, and a fully-capable `ModalShell` that handles
focus trap, scroll lock, Escape-to-close, and Framer Motion animation. The new work is purely
additive — three Firestore helpers, one new modal component, and integration hooks in two existing
editors plus PageManager.

The most structurally significant decision is the inline card-flip metadata form that appears after
each upload. This is a novel UI interaction compared to existing patterns in the codebase and
deserves careful state-machine planning. The multi-select checkout flow for GalleryEditor is the
other high-complexity area: a sticky footer, per-card checkbox state, and a confirmation that
appends items to the block.

The alt-conflict prompt in ImageEditor (D-09) is small in code terms but requires careful focus
management: an inline banner appears inside the block editor after modal close, receives focus, and
must dismiss cleanly.

**Primary recommendation:** Build the Firestore helpers and MediaLibraryModal first (Wave 0 tests
+ data layer), then integrate into ImageEditor/GalleryEditor, then wire PageManager last. Each
wave produces independently testable output.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**D-01:** Media metadata lives in `/cms/media` Firestore collection — separate from `/cms/pages`.
**D-02:** Each document: `storageUrl`, `storagePath`, `title`, `alt`, `uploadedAt` (Timestamp),
`mimeType`, `size` (bytes).
**D-03:** Document ID: `crypto.randomUUID()` at write time.
**D-04:** Query: `orderBy('uploadedAt', 'desc')`, cursor pagination (same pattern as
`listPagesPaginated`). Batch size: Claude's discretion.

**D-05:** Media library modal upload flow — card flips to inline title/alt form after upload;
"Save to Library" commits the Firestore doc; card then shows saved image in grid.
**D-06:** Alt text optional at save time; visible warning shown when left blank. Title optional,
no warning.
**D-07:** Block editor upload flow — same inline title/alt prompt appears within block editor
before URL fills the block field; library record created, URL auto-fills.
**D-08:** Multi-upload from modal — each file gets its own card flip + form; uploads run in
parallel; forms appear as each completes.

**D-09:** Alt conflict prompt — when selecting a library image into a block with existing non-empty
alt: inline banner inside block editor with [Keep mine] / [Replace]. Auto-fills when block alt is
empty (D-10).
**D-11:** Library `title` is library-only — not mapped to any block field.

**D-12:** Modal: ~90vw/85vh, uses `ModalShell` (existing).
**D-13:** Two modal modes: `browse` (no selection callback) and `select`.
**D-14:** ImageEditor: single-select. GalleryEditor: multi-select.
**D-15:** Infinite scroll: `IntersectionObserver` on sentinel element; `listMediaPaginated`.
**D-16:** Upload button inside modal: `image/jpeg,image/png,image/gif,image/webp`; multiple
file selection enabled.

**D-17:** Multi-select mode: visible checkbox on hover/focus per image; sticky footer "N selected"
+ "Add N images" confirm; appends all selected items, closes modal.

**D-18:** Editing existing library items: inline edit panel (card flip or expanded card) for
title/alt; Save writes back to `/cms/media` doc. No block-editor editing of library records.

**D-19:** "Upload Media" button in PageManager opens MediaLibraryModal in browse mode.

### Pattern 1: Firestore Helpers (mirror existing style)

Add to `src/firebase/firestore.js`. Use the same import list already in place (`addDoc` is the
only new Firestore function needed — use it instead of `setDoc` since document IDs are generated
client-side via `crypto.randomUUID()`, meaning `doc(db, 'cms', 'media', id)` + `setDoc` also
works and matches the existing pattern more closely).

```javascript
// Source: src/firebase/firestore.js (existing listPagesPaginated pattern)

export async function addMediaItem(db, item) {
  const id = crypto.randomUUID()
  await setDoc(doc(db, 'cms', 'media', id), {
    ...item,
    uploadedAt: serverTimestamp(),
  })
  return id
}

export async function listMediaPaginated(db, { pageSize = 24, cursor = null } = {}) {
  const col = collection(db, 'cms', 'media')
  const constraints = cursor
    ? [orderBy('uploadedAt', 'desc'), startAfter(cursor), limit(pageSize + 1)]
    : [orderBy('uploadedAt', 'desc'), limit(pageSize + 1)]
  const snap = await getDocs(query(col, ...constraints))
  const hasMore = snap.docs.length > pageSize
  const pageDocs = snap.docs.slice(0, pageSize)
  return {
    items: pageDocs.map(d => ({ id: d.id, ...d.data() })),
    nextCursor: hasMore ? pageDocs[pageDocs.length - 1] : null,
    hasMore,
  }
}

export async function updateMediaItem(db, id, updates) {
  await updateDoc(doc(db, 'cms', 'media', id), updates)
}
```

NOTE: `setDoc` and `updateDoc` are already imported. `doc` is already imported. No new Firestore
imports are needed.

### Pattern 2: MediaLibraryModal Structure

```javascript
// Simplified skeleton — Source: derived from ModalShell.js prop API
export function MediaLibraryModal({ open, mode, onSelect, triggerRef, onClose }) {
  // mode: 'browse' | 'select-single' | 'select-multi'
  // onSelect: (item) => void for single, (items[]) => void for multi
  const headingId = 'media-library-heading'
  return (
    <ModalShell
      open={open}
      labelId={headingId}
      triggerRef={triggerRef}
      onClose={onClose}
      backdropStyle={{ zIndex: 1100 }}  // above standard modals (z:1000)
    >
      <div className="jeeby-cms-media-library">
        <header className="jeeby-cms-media-library-header">
          <h2 id={headingId}>Media Library</h2>
          <button onClick={onClose} aria-label="Close media library">…</button>
        </header>
        {/* upload zone, grid, sentinel, sticky footer */}
      </div>
    </ModalShell>
  )
}
```

The existing `.jeeby-cms-modal-card` CSS sets `max-width: 480px`. The media library needs
~90vw/85vh (D-12). The implementor must either:
- Add a modifier class (e.g., `jeeby-cms-modal-card--full`) to cms.css
- Or pass a `backdropStyle` override — but backdropStyle only covers the backdrop, not the card

The correct approach is a modifier class on the card that overrides the max-width. ModalShell
renders `.jeeby-cms-modal-card` as the card div, so the child component cannot directly apply a
class to it. Two implementation options exist:
- Option A: Add an optional `cardClassName` prop to ModalShell (non-breaking, clean)
- Option B: Wrap ModalShell content in a full-bleed inner div and set the grid dimensions on it, accepting that the outer card is narrower

Option A is architecturally cleaner and is the recommended approach.

### Pattern 3: Infinite Scroll with IntersectionObserver

```javascript
// Source: MDN IntersectionObserver API (browser built-in)
const sentinelRef = useRef(null)

useEffect(() => {
  if (!hasMore || loading) return
  const observer = new IntersectionObserver(
    ([entry]) => { if (entry.isIntersecting) loadNextPage() },
    { threshold: 0 }
  )
  const el = sentinelRef.current
  if (el) observer.observe(el)
  return () => { if (el) observer.unobserve(el) }
}, [hasMore, loading, loadNextPage])

// In render:
// <div ref={sentinelRef} aria-hidden="true" />
```

The sentinel must be placed AFTER the last grid item. It should be `aria-hidden="true"` — it has
no semantic meaning.

### Pattern 4: Card Flip Animation (Framer Motion)

Framer Motion's `AnimatePresence` + `motion.div` with `rotateY` creates a genuine card flip. A
simpler approach that matches the project's "purposeful, not decorative" principle: crossfade with
`AnimatePresence` mode="wait". The flip should respect `useReducedMotion()` — when true, skip
the animation entirely (duration 0.01 fallback, same pattern as ModalShell).

```javascript
// Source: framer-motion AnimatePresence docs
const reduced = useReducedMotion()
<AnimatePresence mode="wait">
  {isEditing ? (
    <motion.div key="form" initial={{ opacity: 0 }} animate={{ opacity: 1 }} exit={{ opacity: 0 }}
      transition={{ duration: reduced ? 0.01 : 0.2 }}>
      {/* title/alt form */}
    </motion.div>
  ) : (
    <motion.div key="image" initial={{ opacity: 0 }} animate={{ opacity: 1 }} exit={{ opacity: 0 }}
      transition={{ duration: reduced ? 0.01 : 0.2 }}>
      {/* thumbnail */}
    </motion.div>
  )}
</AnimatePresence>
```

### Pattern 5: Post-Upload Inline Prompt State Machine

Each media card in the modal has these states:
1. `idle` — shows thumbnail (or empty slot)
2. `uploading` — shows progress bar
3. `pending-meta` — upload complete, shows title/alt form ("card flip")
4. `saved` — shows thumbnail + saved indicator

In ImageEditor/GalleryEditor the post-upload state is inline within the block editor:
1. Upload completes → `pending-meta` state replaces the normal URL/upload controls
2. User fills title/alt (alt warning appears if left blank) and clicks "Save to Library"
3. `addMediaItem(db, {...})` called → `src` set on block data → returns to normal edit mode

### Anti-Patterns to Avoid

- **Storing Firestore DocumentSnapshots in React state:** Pass the snapshot as `nextCursor` to
  `listMediaPaginated` but do not store snapshots as React state. Store the cursor in a ref or
  as a separate state variable.
- **Using `listPages` (non-paginated) for media:** The media library will grow over time and
  must use `listMediaPaginated` from the start.
- **Calling `addMediaItem` before upload is confirmed:** The library record must not be written
  until after the user clicks "Save to Library" or "Confirm" — not immediately after the file
  upload resolves. This is explicit in D-05 and D-07.
- **Closing the modal before the user confirms metadata:** The "Save to Library" step happens
  inside the modal; closing before confirming would create orphan Storage files with no Firestore
  record. D-05 explicitly sequences: upload → card flip → Save → then the item appears in grid.
- **Inline styles for layout in new components:** Established rule (Phase 8): CSS classes only,
  no inline styles except runtime-dynamic values (e.g., `style={{ width: '${pct}%' }}`).

## Common Pitfalls

### Pitfall 1: ModalShell card size constraint

**What goes wrong:** MediaLibraryModal inherits `.jeeby-cms-modal-card { max-width: 480px }` from
the existing CSS, rendering as a narrow dialog instead of the 90vw/85vh grid specified in D-12.

**Why it happens:** ModalShell hard-codes `className="jeeby-cms-modal-card"` on the inner div.
There is no existing mechanism to pass a modifier class.

**How to avoid:** Add an optional `cardClassName` prop to ModalShell. The planner must include
this as an explicit task before building MediaLibraryModal.

**Warning signs:** If the modal renders at 480px wide, this pitfall has been hit.

### Pitfall 2: Orphan Storage files from abandoned metadata prompts

**What goes wrong:** A user uploads a file, sees the title/alt form, then closes the modal or
navigates away without clicking "Save to Library". The file exists in Firebase Storage with no
Firestore record.

**Why it happens:** Upload and metadata entry are two distinct steps. If the session ends between
them, the Firestore write never happens.

**How to avoid:** This is an accepted v1 tradeoff (no storage deletion from UI per D-18 and
deferred scope). Document clearly in code comments. For the modal, warn the user if they attempt
to close while a card is in `pending-meta` state (similar to the `uploadCountRef` guard in
GalleryEditor).

**Warning signs:** Files accumulating in `cms/media/images/` with no corresponding Firestore
document.

### Pitfall 3: `uploadedAt` field missing Firestore composite index

**What goes wrong:** `listMediaPaginated` calls `orderBy('uploadedAt', 'desc')` + `limit()`. If
Firestore does not have an index for this field, the query throws a runtime error in production
(though it may work in emulator/development).

**Why it happens:** Single-field `orderBy` on a server timestamp generally does not need a
composite index — Firestore creates single-field indexes automatically. However, if any future
query adds a `where()` clause (e.g., filter by `mimeType`), a composite index will be required.

**How to avoid:** The current paginated query (no `where()`) does not need a manual index. Note
this for future filtering work. Verify in Firebase Console after first deploy.

**Warning signs:** Firestore throws `FAILED_PRECONDITION: The query requires an index` at runtime.

### Pitfall 4: Focus management when modal closes after single-select

**What goes wrong:** When a user selects an image in single-select mode, the modal closes and
focus returns to the element referenced by `triggerRef`. If `triggerRef` is stale (e.g., the
"Select from Library" button was inside a component that re-rendered), focus is lost and jumps to
the document body — a WCAG 2.4.3 failure.

**Why it happens:** React re-renders can replace DOM nodes; a ref to a button from before a
re-render may point to a detached node.

**How to avoid:** Use `useCallback` for the handler that opens the library modal so the trigger
button's ref remains stable. The ref is already the established pattern (same as CreatePageModal,
DeletePageModal).

**Warning signs:** After modal closes, keyboard focus moves to body instead of the trigger button.

### Pitfall 5: Alt conflict prompt and focus order in ImageEditor

**What goes wrong:** The alt conflict prompt (D-09) appears inline below the URL input after the
modal closes. If it appears without receiving focus, keyboard users will not know it exists —
they may have already moved past the input in tab order.

**Why it happens:** Dynamic content appearing in the DOM does not automatically receive focus.

**How to avoid:** On appearance, call `focus()` on the first interactive element in the conflict
banner (the "Keep mine" or "Replace" button), or use `role="alert"` to announce it via live
region. Since the user must make a choice, moving focus to the button is the better UX.

**Warning signs:** Screen reader user never hears the conflict prompt announcement.

### Pitfall 6: `setDoc` vs Firestore collection path

**What goes wrong:** `addMediaItem` uses `doc(db, 'cms', 'media', id)` but existing helpers in
`firestore.js` use `doc(db, 'pages', slug)` — the `cms` subcollection segment is correct for
media but easy to accidentally write as `doc(db, 'media', id)` (missing `'cms'`).

**Why it happens:** Inconsistency — pages are at root `pages/` collection, media is at `cms/media/`.

**How to avoid:** The CONTEXT.md explicitly specifies `/cms/media` as the collection. One of the
existing firestore.test.js tests even checks that `listPages` uses `collection(db, 'cms', 'pages')`
— apply the same pattern. Add a test that reads the source for `collection(db, 'cms', 'media')`.

**Warning signs:** `listMediaPaginated` returns 0 results even after uploading, because documents
were written to `media/` instead of `cms/media/`.

### Pitfall 7: GalleryEditor multi-select state when modal unmounts

**What goes wrong:** If the library modal is unmounted (not just hidden) between open/close cycles,
checkbox state (`selected` Set) is lost. This is fine — desired, even. The problem arises if the
modal mounts once, receives a batch of checked items, gets closed and re-opened, and the previous
selection is still visible.

**Why it happens:** If `selected` state is owned by the parent (GalleryEditor) rather than the
modal, it persists across open/close.

**How to avoid:** Checkbox state belongs inside the modal component, not the parent. Reset on each
`open=true` transition using a `key` prop or a `useEffect` that clears state when `open` changes
from false to true.

## Project Constraints (from CLAUDE.md)

Directives the planner must verify all tasks comply with:

1. WCAG AA accessibility enforced throughout — every new interactive element needs: visible label,
   keyboard operability, focus management, and appropriate ARIA role/state.
2. Semantic HTML before ARIA — use `<button>` not `<div role="button">`, `<input type="checkbox">`
   not custom checkbox widgets.
3. One H1 per admin page — MediaLibraryModal uses H2 for its heading (not H1).
4. Focus managed on modal open (auto-focus first focusable), close (return to triggerRef), and
   after dynamic content appears (alt conflict prompt).
5. Live regions for dynamic updates — upload progress announced via `aria-live="polite"`;
   errors via `role="alert"`.
6. Text contrast 4.5:1, UI component contrast 3:1 — apply CSS custom properties; do not
   introduce hardcoded colors.
7. CSS classes only, no inline styles except runtime-dynamic values — established in Phase 8.
8. All new CSS classes prefixed `jeeby-cms-*`.
9. "use client" directive at top of every admin component file.
10. No new npm dependencies — all libraries are peer deps already.
11. Modals trap focus (ModalShell handles this automatically — use it).
12. Every destructive or irreversible action needs confirmation — note: "Save to Library" is
    additive (not destructive), no confirmation needed; alt conflict prompt's [Replace] is
    reversible since it only affects the block's local alt field.

## Open Questions

1. **ModalShell `cardClassName` prop: add or workaround?**
   - What we know: ModalShell does not currently accept a modifier class for the card element.
     The media library needs 90vw/85vh card, not the existing 480px max-width.
   - What's unclear: Whether to modify ModalShell (adding `cardClassName` prop) or use a
     different nesting approach.
   - Recommendation: Add `cardClassName` prop to ModalShell — clean, non-breaking, one-line change.
     Plan 01 (Wave 0) should include this as a prerequisite task.

2. **Firestore collection path: `cms/media` or `media`?**
   - What we know: CONTEXT.md D-01 specifies `/cms/media`. The existing `firestore.test.js`
     has a test that checks `collection(db, 'cms', 'pages')` — confirming the `cms` prefix is
     intentional across the codebase.
   - What's unclear: Nothing. This is resolved — use `collection(db, 'cms', 'media')`.
   - Recommendation: Locked. Use `doc(db, 'cms', 'media', id)`.

3. **Batch size for listMediaPaginated**
   - What we know: CONTEXT.md suggests 20-24, defers to Claude.
   - What's unclear: Optimal for the grid layout.
   - Recommendation: 24 — divides cleanly into 3 and 4 column grids (8 or 6 rows per batch).
     Balanced between network efficiency and scroll feel.

4. **`pending-meta` state during block editor upload — where in the UI?**
   - What we know: D-07 says the inline title/alt prompt appears within the block editor before
     the URL fills. ImageEditor currently uses an `uploadControls` JSX variable shared across
     edit modes.
   - What's unclear: Whether to add a 5th state to ImageEditor's state machine or to render
     the prompt as an overlay on the existing edit mode.
   - Recommendation: Add a `pendingMeta` state (analogous to `pending-meta` card state in modal).
     The `uploadControls` variable becomes conditional: when `pendingMeta` is set, show the
     title/alt form instead of the URL/upload controls. The Done button remains disabled until
     the user saves or explicitly skips.
